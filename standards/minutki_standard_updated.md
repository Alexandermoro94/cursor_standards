# Minutki Standard

## Цель стандарта

Данный документ будет использоваться менеджерами для подготовки минуток после звонков, где по разделам будут описаны задачи с DoD, последовательностью задач, блокерами и дополнительными комментариями.

## Input для подготовки минуток

Для получения качественных минуток, соответствующих стандарту, необходимо иметь следующий входной материал:

**Input (общий для всех минуток):**
- Записи/расшифровка встречи или звонка (текстовый документ с описанием обсужденных тем, решений, задач, например: "Расшифровка встречи от 3 декабря: обсуждали титутлы, обмен опытом, карту отчуждения")
- Контекст встречи (информация о целях встречи, участниках, обсуждаемых проектах, например: "Встреча по планированию активностей на декабрь, участники: Лера, Саша, обсуждали титутлы и обмен опытом")
- Список задач, которые были обсуждены (структурированный список задач с кратким описанием, например: "Обновить титутлы", "Создать таблицу по обмену опытом", "Доделать карту отчуждения")
- Информация о дедлайнах и приоритетах (если обсуждались, например: "Таблица JTBD нужна к среде", "Пост с титутлами - приоритет высокий")
- Связь с картой целей (понимание, как задачи связаны с бизнес-целями и outcome, например: "Задачи по титутлам связаны с outcome 'Увеличение вовлеченности участников'")
- Эталоны и референсы (если есть примеры похожих задач или эталонные артефакты, например: "Шаблон анализа кастдевов", "Пример хороших минуток")
- Информация об ответственных (кто будет выполнять задачи, например: "Лера - подготовка таблиц, Саша - ревью")

**Output (общий для всех минуток):**
- Структурированные минутки с задачами по разделам (документ в формате Markdown или текстовый файл с иерархией разделов: Организационные вопросы, Монетизация, Активности, Эдвайзеринг, Дополнительно, На будущее)
- Каждая задача содержит: контекст (ситуация + проблема), DoD с проверкой Output и Gap Analysis, детальный Output с параметрами проверки, ответственного, дедлайн, артефакты
- Минутки готовы к использованию командой для выполнения задач (все задачи имеют полный набор обязательных элементов, проверены по чеклисту качества)

### Примеры Input для конкретных задач (из хорошего примера)

#### Пример 1: Задача "Связаться с Машей Торопыгиной по обновлению титутлов"

**Input для этой задачи:**
- Информация о задаче (краткое описание, например: "Нужно связаться с Машей Торопыгиной по обновлению титутлов")
- Контекст обсуждения (описание ситуации, например: "Планируем обновить титутлы, рассматриваем Машу как дизайнера для помощи")
- Неясности (что не понятно, например: "Неясно, актуально ли к ней приходить с задачей по титутлам и в каком формате")
- Контактные данные Маши (способ связи, например: "Telegram: @masha_toropygina, email: masha@example.com")
- Текущее состояние (есть ли уже переписка или договоренности, например: "Нет предыдущей переписки с Машей по титутлам")
- Связь с outcome (как задача связана с бизнес-целями, например: "Увеличение вовлеченности участников через систему мотивации")
- Ответственный (кто будет выполнять, например: "Лера")
- Дедлайн (если указан, например: "не указан (—)")

**Output для этой задачи (из хорошего примера):**
- Заметка с итогом договоренностей с Машей
- Ссылка на заметку
- При необходимости скрин переписки

#### Пример 2: Задача "Начать карту отчуждения по минуткам для регулярных встреч"

**Input для этой задачи:**
- Информация о задаче (краткое описание, например: "Нужна карта отчуждения по минуткам для встреч")
- Контекст проблемы (описание ситуации, например: "У Леры уходит много времени на подготовку минуток, нужно это отдать новому стажеру")
- Текущий процесс (описание того, как сейчас готовятся минутки, например: "Лера готовит минутки вручную: слушает запись встречи, выписывает задачи, оформляет по стандарту, занимает 2-3 часа")
- Примеры минуток (ссылки на примеры, например: "Ссылка на минутки от 3 декабря, минутки от 10 ноября")
- Требования к карте (что должно быть в карте, например: "Этапы процесса подготовки минуток, точки возможного отчуждения, где стажер может потеряться")
- Связь с outcome (как задача связана с бизнес-целями, например: "Освобождение времени Леры для стратегических задач")
- Ответственный (кто будет выполнять, например: "Лера")
- Дедлайн (если указан, например: "не указан (—)")

**Output для этой задачи (из хорошего примера):**
- Первый драфт карты отчуждения по минуткам для встреч
- Ссылка на карту
- Список использованных минуток

#### Пример 3: Задача "Таблица по обмену опытом (гипотезы, под ревью Саши)"

**Input для этой задачи:**
- Информация о задаче (краткое описание, например: "Нужна таблица с гипотезами по обмену опытом для ревью Саши")
- Контекст (описание ситуации, например: "Есть разные идеи по форматам обмена опытом, сейчас они разбросаны")
- Проблема (что не работает, например: "Нельзя быстро посмотреть все гипотезы и дать по ним ревью")
- Текущее состояние (где хранятся идеи, например: "Идеи разбросаны в разных местах: в заметках, в чатах, в документах")
- Список гипотез (все текущие идеи, например: "Стрим минутки, формат обмена кейсами, мастер-классы от участников")
- Формат таблицы (требования к структуре, например: "Столбцы: Гипотеза №, Название гипотезы, Описание, Статус, Комментарии")
- Доступы (где создать и как предоставить доступ, например: "Создать в Google Sheets, предоставить доступ Саше на просмотр")
- Связь с outcome (как задача связана с бизнес-целями, например: "Увеличение ценности сообщества через обмен знаниями")
- Ответственный (кто будет выполнять, например: "Лера (подготовка), Саша (просмотр)")
- Дедлайн (если указан, например: "не указан (—)")

**Output для этой задачи (из хорошего примера):**
- Таблица гипотез по обмену опытом
- Ссылка на таблицу
- Структура таблицы с описанием столбцов
- Параметры проверки

#### Пример 4: Задача "Пост с титутлами за ноябрь выложить"

**Input для этой задачи:**
- Информация о задаче (краткое описание, например: "Нужно выложить пост с титутлами за ноябрь")
- Контекст (описание ситуации, например: "В сообществе есть система титутлов, но нет отдельного понятного поста про нее")
- Проблема (что не работает, например: "Участникам сложно быстро понять, какие титутлы есть и за что они даются")
- Список титутлов (актуальный список, например: "Титул 'Герой месяца', 'Активный участник', 'Эксперт по конверсии'")
- Правила получения (описание правил, например: "Титул дается за активное участие в обсуждениях, за помощь другим участникам, за публикацию кейсов")
- Формат поста (требования к формату, например: "Публиковать в основной группе ВКонтакте, стиль: информативный, без картинок")
- Референсы (примеры предыдущих постов, например: "Пост с титутлами за октябрь 2025")
- Связь с outcome (как задача связана с бизнес-целями, например: "Прозрачность системы мотивации увеличивает вовлеченность")
- Ответственный (кто будет выполнять, например: "Лера")
- Дедлайн (если указан, например: "не указан (—)")

**Output для этой задачи (из хорошего примера):**
- Опубликованный пост с титутлами
- Ссылка на пост
- Структура поста (заголовок, список титутлов, правила получения, призыв к действию)
- Параметры проверки
- Черновик поста (если был)

#### Пример 5: Задача "Таблица анализа JTBD (доделать)"

**Input для этой задачи:**
- Информация о задаче (краткое описание, например: "Нужно доделать таблицу анализа JTBD")
- Контекст (описание ситуации, например: "Есть расшифровки разговоров и потребности участников, но они не сведены в единую структуру по JTBD")
- Проблема (что не работает, например: "Нельзя быстро увидеть для каждой потребности весь сценарий ее возникновения и попытки закрытия")
- Расшифровки разговоров (ссылки на расшифровки, например: "Ссылка на расшифровку разговора с участником А от 15 ноября, расшифровка разговора с участником Б от 20 ноября")
- Текущее состояние таблицы (ссылка и статус, например: "Ссылка на таблицу 'JTBD анализ', заполнено 2 потребности из 5, остальные нужно добавить")
- Структура таблицы (требования к столбцам, например: "Столбцы: Потребность №, Потребность, Триггер, Сценарий, Выгода, Как раньше закрывал, Блокеры в сценарии/старом способе, Была ли попытка закрыть в коммьюнити, Блокеры в попытке в коммьюнити, Получил ли желаемый результат")
- Дедлайн (конкретная дата, например: "к ближайшей среде (6 декабря 2025)")
- Связь с outcome (как задача связана с бизнес-целями, например: "Понимание потребностей участников для улучшения продуктов и сервисов")
- Ответственный (кто будет выполнять, например: "Лера")

**Output для этой задачи (из хорошего примера):**
- Таблица анализа JTBD с заполненными данными
- Ссылка на таблицу
- Структура таблицы с описанием всех столбцов
- Параметры проверки
- Ссылки на расшифровки, которые использовались

### Принципы формирования Input для задачи

При подготовке Input для каждой задачи необходимо собрать следующие данные (каждый пункт должен содержать конкретные примеры в скобках):

1. **Информация о задаче:**
   - Краткое описание задачи (что нужно сделать, например: "Создать таблицу с гипотезами")
   - Контекст обсуждения (почему эта задача возникла, например: "На встрече обсуждали форматы обмена опытом")

2. **Текущее состояние:**
   - Где сейчас находится информация/данные для задачи (например: "Идеи разбросаны в заметках и чатах")
   - Что уже сделано (если задача "доделать", например: "Заполнено 2 потребности из 5 в таблице JTBD")
   - Какие есть материалы (документы, ссылки, данные, например: "Ссылка на расшифровку разговора от 15 ноября")

3. **Требования и ограничения:**
   - Дедлайны (если есть, например: "к ближайшей среде (6 декабря 2025)")
   - Формат результата (таблица, пост, документ и т.д., например: "Таблица в Google Sheets с 5 столбцами")
   - Доступы (кто должен иметь доступ к результату, например: "Саша должен иметь доступ на просмотр")

4. **Связь с бизнес-целями:**
   - Как задача связана с outcome из карты целей (например: "Увеличение вовлеченности участников через систему мотивации")
   - Почему эта задача важна (например: "Прозрачность системы мотивации увеличивает вовлеченность")

5. **Референсы и эталоны:**
   - Примеры похожих задач (если есть, например: "Минутки от 3 декабря как референс")
   - Шаблоны или эталоны (если применимо, например: "Шаблон анализа кастдевов для структуры таблицы")

**Важно:** Каждый пункт Input должен содержать конкретные примеры в скобках, как показано в примерах выше. Это помогает понять, какие именно данные нужны для формирования качественной минутки.

## Структура минуток

### 3.1 Соблюдается иерархия

Сперва разделы, внутри разделов отдельные задачи (тикеты), внутри задачи подпункты:
- Контекст (обязателен)
- Definition of Done
- Output
- Ответственный
- Дедлайн
- Артефакты

#### Список текущих разделов (верхний уровень):

1. Организационные вопросы — доступы, структура, расписание, регламенты, встречи, документация и т.п.
2. Монетизация — задачи касающиеся воронки по подключению новых участников, подписок и донатов.
3. Активности — контент-план, подготовка/оформление/публикация материалов, каналы, расписание, теги, метрики CTR/ER, разборы кейсов участников, генерация идей для обмена опытом.
4. Эдвайзеринг — клиентские задачи, рекомендации, подготовка/проведение сессий, сценарии участников (без публикаций).
5. Дополнительно — прочие задачи, не попавшие в основные разделы.
6. На будущее — бэклог идей и задач на последующие итерации (пункты в этом разделе можно оформлять не по шаблону оформления задач, а просто тезисом. Например "новый титул за принятие участие в активности").

### 3.2 Пример оформленной задачи (шаблон, числовой формат)

```
1. Организационные вопросы

1.1. Настройка ИИ-агента для загрузки материалов по NDA в именную папку

Контекст:
Ситуация: Когда пользователь хочет попасть в эдвайзеринг, он отправляет на почту сканы своего паспорта и подписанного NDA. Сейчас эти материалы сохраняются вручную, что занимает время и может привести к ошибкам при поиске документов конкретного пользователя.

Проблема: Отсутствует автоматизация процесса сохранения NDA, что увеличивает время обработки заявок и риск потери документов.

Definition of Done:
- [ ] В n8n рамках проекта heroesofconversion.space@gmail.ru настроен флоу
- [ ] Этот флоу при получении триггера (получение письма с сканами) сохраняет их в именной папке
- [ ] Проведено тестирование на 5+ реальных письмах с NDA
- [ ] Проверка Output: ссылка на папку с NDA работает, папка содержит сканы паспорта и подписанного NDA для каждого отправителя
- [ ] Gap Analysis: сравнение с ручным процессом (Expected: автоматическое сохранение, Actual: [заполняется при проверке], Gap: [заполняется при проверке], Decision: [fix/ok], Evidence: ссылки на тестовые письма и результаты)

Output:
Финальный артефакт: автоматически созданная именная папка отправителя в общей папке с NDA
Ссылка: [ссылка на папку с NDA]
Структура: для каждого отправителя создается отдельная папка с именем отправителя, внутри которой находятся:
- Скан паспорта (файл с расширением .pdf или .jpg)
- Подписанный NDA (файл с расширением .pdf)
Параметры проверки:
- Имя папки соответствует имени отправителя из письма
- В папке присутствуют оба файла (паспорт и NDA)
- Файлы доступны для просмотра и скачивания

Ответственный: Лера

Дедлайн: 9 октября 2025

Артефакты:
- Ссылка на папку с NDA
- Ссылка на проект в n8n
```

Пример для задачи с таблицей:

```
3. Активности

3.1. Таблица анализа кастдевов (3 разговора)

Контекст:
Ситуация: Проведены три интервью по кастдеву для понимания потребностей участников сообщества. Результаты интервью записаны в разных форматах и не структурированы.

Проблема: Без структурирования результатов невозможно провести анализ и выявить общие паттерны потребностей участников.

Definition of Done:
- [ ] Для каждого разговора заполнены поля: потребность, сценарий, желаемый ауткам, как закрывает сейчас, какие проблемы
- [ ] Таблица приведена к единому формату
- [ ] Проверка Output: ссылка на таблицу работает, все столбцы заполнены, данные структурированы
- [ ] Artefact Comparison Challenge: сравнение с эталоном (шаблон анализа кастдевов) - структура эталона соответствует, критерии эталона выполнены
- [ ] Gap Analysis: Expected: таблица с 3 строками, все столбцы заполнены, формат соответствует шаблону; Actual: [заполняется при проверке]; Gap: [заполняется при проверке]; Decision: [fix/ok]; Evidence: ссылка на таблицу и шаблон
- [ ] Все три разговора проанализированы и задокументированы

Output:
Финальный артефакт: таблица "НОС Custdev November 2025" с результатами анализа 3 разговоров
Ссылка: [ссылка на таблицу]
Структура таблицы (столбцы):
- Разговор № (номер разговора: 1, 2, 3)
- Потребность (описание потребности участника)
- Сценарий (описание сценария использования)
- Желаемый ауткам (что хочет получить участник)
- Как закрывает сейчас (текущее решение проблемы)
- Какие проблемы (выявленные проблемы текущего решения)
Параметры проверки:
- Все 3 строки заполнены (по одной на каждый разговор)
- Все столбцы содержат данные (нет пустых ячеек)
- Данные структурированы и понятны
- Формат соответствует шаблону анализа кастдевов

Ответственный: Лера

Дедлайн: 3 декабря 2025

Артефакты:
- Ссылка на таблицу "НОС Custdev November 2025"
- Шаблон анализа кастдевов
```

Пример хороших минуток (полный формат):

### 3.3 Критические различия: хороший vs плохой результат

**Что было упущено в плохом результате и что нужно исправить:**

#### 1. Definition of Done
**Плохой результат:**
- DoD содержит только базовые чекбоксы действий
- Нет проверки Output с параметрами
- Нет Gap Analysis

**Хороший результат:**
- DoD содержит чекбоксы действий
- **Обязательно** есть пункт "Проверка Output" с описанием параметров и ожидаемых значений
- **Обязательно** есть Gap Analysis с форматом: Expected, Actual, Gap, Decision, Evidence

**Что изменить:**
- Добавить в DoD обязательный пункт проверки Output для каждой задачи
- Добавить в DoD обязательный Gap Analysis для каждой задачи (даже если Actual/Gap/Decision заполняются при проверке, Expected должен быть указан сразу)

#### 2. Output
**Плохой результат:**
- Output содержит только финальный артефакт и ссылку
- Нет параметров проверки
- Для таблиц нет детального описания столбцов с назначением

**Хороший результат:**
- Output содержит финальный артефакт, ссылку, структуру
- **Обязательно** есть параметры проверки с конкретными ожидаемыми значениями
- Для таблиц есть детальное описание каждого столбца с назначением

**Что изменить:**
- Всегда добавлять в Output раздел "Параметры проверки" с конкретными ожидаемыми значениями
- Для таблиц обязательно описывать каждый столбец с его назначением
- Для документов/процессов обязательно описывать структуру

#### 3. Контекст
**Плохой результат:**
- Контекст есть, но менее детальный
- Не всегда четко разделены Ситуация и Проблема

**Хороший результат:**
- Контекст содержит четко разделенные Ситуацию и Проблему
- Ситуация описывает текущее состояние через JTBD-подход

**Что изменить:**
- Всегда четко разделять Ситуацию и Проблему в контексте
- Формулировать Ситуацию через JTBD-подход (Роль-Контекст-Потребность)

### 3.4 Дополнительные требования

#### Контекст

- **Структура контекста:**
  1. **Ситуация** — описание текущего состояния, процесса или обстоятельств
  2. **Проблема** — какая проблема возникла или может возникнуть в описанной ситуации

- **Контекст должен говорить о связке с outcome из карты целей** — как текущая задача связана с ним
- **Контекст должен быть сформулирован через JTBD**, используя паттерн "Роль-Контекст-Потребность" (в описании ситуации)

#### Output

- **Output должен содержать (ОБЯЗАТЕЛЬНО):**
  - **Описание финального артефакта** (что конкретно получится) - обязателен для всех задач
  - **Ссылку на артефакт** (если применимо) - обязательна, если артефакт имеет ссылку
  - **Структуру артефакта** (для документов, таблиц, процессов) - обязательна для всех задач
  - **Столбцы** (для таблиц) с описанием назначения каждого столбца - обязательны для таблиц
  - **Параметры проверки** (что должно быть в артефакте, какие значения ожидаются) - обязательны для всех задач

- **Для таблиц обязательно указывать:**
  - Название таблицы
  - Список столбцов с описанием
  - Ожидаемое количество строк (если известно)
  - Формат данных в каждом столбце

- **Для документов/файлов указывать:**
  - Название документа
  - Структуру (разделы, подразделы)
  - Формат файла

- **Для процессов/систем указывать:**
  - Описание процесса
  - Входные и выходные данные
  - Ключевые этапы

#### Definition of Done

- **Definition of Done должен содержать (ОБЯЗАТЕЛЬНО):**
  - Чеклист выполнения ключевых требований задачи
  - **Чеклист проверки Output** с описанием параметров и ожидаемых значений (обязателен для всех задач)
  - **Gap Analysis** (обязателен для всех задач, формат: Expected, Actual, Gap, Decision, Evidence)
  - Критерии проверки качества результата

- **Принципы составления DoD (на основе from_the_end.md):**
  - Фиксировать только самые важные требования
  - Опускать технические нюансы
  - Включать проверку соответствия Output ожиданиям
  - Указывать параметры проверки с конкретными ожидаемыми значениями
  - При необходимости включать сравнение с эталоном (Artefact Comparison Challenge)

- **Artefact Comparison Challenge (если применимо):**
  При наличии эталона для сравнения в DoD должно быть указано:
  - **Эталонный артефакт:** ссылка на идеальный пример или описание эталона
  - **Структура эталона:** описание структуры и содержания эталона
  - **Критерии эталона:** конкретные требования к качеству, которым должен соответствовать результат
  - **Контекст эталона:** условия и ограничения, при которых эталон применим

- **Gap Analysis (ОБЯЗАТЕЛЬНО для всех задач):**
  В DoD должно быть указано:
  - **Expected:** конкретное описание ожидаемого результата (заполняется при создании задачи)
  - **Actual:** фактический результат после выполнения (заполняется при проверке)
  - **Gap:** конкретные различия между ожидаемым и фактическим (заполняется при проверке)
  - **Decision:** решение (fix/ok) после анализа gap (заполняется при проверке)
  - **Evidence:** ссылки на скриншоты, тесты и результаты проверки (заполняется при проверке)

- **Административные пункты не включать в чеклист качества** (например, "ссылка добавлена в репозиторий материалов"). Такие действия оформлять как отдельные задачи проекта "Организация" либо как артефакты в DoD без чекбоксов качества.

Правила форматирования (обязательно):
- Использовать числовую иерархию разделов: 1., 1.1., 1.1.1 при необходимости.
- В качестве верхнего уровня использовать разделы: «Организационные вопросы», «Монетизация», «Активности», «Эдвайзеринг», «Дополнительно», «На будущее» (при наличии соответствующих задач).
- Не использовать эмодзи в заголовках и тексте минуток.
- Не писать раздел «Блокеры и вопросы», если блокеров нет.
- Не использовать горизонтальные разделители '---'.
- Минимизировать пустые строки (не более одной подряд).
- В Definition of Done фиксировать только самые важные требования, опускать технические нюансы.
- **Обязательно** включать в DoD пункт "Проверка Output" и Gap Analysis для каждой задачи.
- **Обязательно** включать в Output раздел "Параметры проверки" для каждой задачи.
